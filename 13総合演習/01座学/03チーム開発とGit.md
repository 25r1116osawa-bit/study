# チーム開発とGit
## 1. チーム開発におけるGitの役割
チームワークを「共通の目標に向かって複数の人間が共に行動し、一定の成果を上げること」と定義しています。ITプロジェクトにおいて、成果物（コード）を統合し、共通目標を達成するための「**プロジェクトルール**」の核となるのがGitです。

*   **体制とルールの遵守**: 誰がどの作業を担当し、どのツール（Git等）を使って成果物を管理するかというルールを守ることが、メンバーとしての信頼獲得の近道です。
*   **作業の標準化**: Gitを用いた定型的なワークフローを実践することで、作業を効率化し、生産性を向上させます。

## 2. 枝分かれ（ブランチ戦略）によるリスク管理

*   **プロジェクトの5つのプロセス**: プロジェクトには「立ち上げ・計画・実行・コントロール・終結」の5つのプロセスがあります。ブランチ（作業の枝分かれ）を作ることは、メインの成果物に影響を与えずに「実行」し、不備があれば「コントロール（是正）」するための重要な仕組みです。
*   **戦略の重要性**: 「どのブランチで作業し、どこに統合するか」というルールを明確にすることは「**5W4Hの計画**」（何を、どのように）を確認することに他なりません。

## 3. コードレビュー文化と「レビューの3原則」
Git上で行われる「プルリクエスト（コードレビュー）」は*品質確保の原則**を具体化したものです。

1.  **セルフチェックの原則**: 自分のコードに責任を持ち、レビューを依頼する前に自分で確認する。
2.  **クイックチェックの原則**: 作業が完了したら直ちにレビューを依頼し、フィードバックを早期に受ける。
3.  **クロスチェックの原則**: 複数の視点で確認し、バグの混入を防ぐとともに、チーム内での技術共有（相互理解）を図る。

これにより、個人のミスが全体の品質を崩壊させる「**ドミノ倒し**」を未然に防ぎます。

### 4. コンフリクト発生時のコミュニケーション
コードの衝突（コンフリクト）は技術的な問題に見えますが、その解決には「**主体的なコミュニケーション**」が不可欠です。

*   **問題の識別と意思決定**: コンフリクトが起きた際、感情的にならずに「何が問題か（事実）」を把握し、どちらの変更を優先すべきかチームで「意思決定」します。
*   **悪い報告は早めに**: 解決に時間がかかりそうな場合は、「納期（Delivery）」への影響を考慮し、隠さず早めにリーダーへ報告（報連相）します。
*   **マインドセット**: 相手を尊重し、相手の立場に立って物事を考えることが、円滑な課題解決と良好な人間関係に繋がります。
