# プログラミング総合演習準備
## チーム発足と開発における推奨ロール構成

- プロジェクトチーム名を決定して下さい。
- ※兼業可能です。
- ※全員コーディングは行って下さい。

| ロール名 | 主な任務（責任範囲） | 講師（顧客）からの期待 |
| --- | --- | --- |
| **プロジェクトリーダー (PL)** | 全体の進捗管理、タスクの割り振り、最終納品判断。 | チームとしての意思決定の窓口。 |
| **Git/ドキュメント管理者** | リポジトリ構成の維持、`doc/`内の整理、マージ作業。 | 「正解のコード」と「証拠書類」を守る門番。 |
| **開発リード** | 実装の中心。技術的な詰まりどころの解消。 | 顧客要望を正しくコードに変換する職人。 |
| **QA（品質保証）担当** | テスト項目の作成、バグの発見、QA表の更新。 | 「動かない」「仕様と違う」を許さない最後の砦。 |

## 演習準備

### 1. GitHub：納品・レビュー環境の構築

* **リポジトリを作成する**
* プロジェクトチーム名とクラス名(25R1116)を含めて下さい。
* **例：** `25R1116_TeamName_E-Commerce-Project`
* リポジトリ内に doc/ ディレクトリを作成してください。

* **講師とのコラボレーション**
* 講師をリポジトリの「Collaborator」または「Organizationメンバー」として招待する。

### 2. Slack：コミュニケーション基盤の整備

* **グループ構成:**
* 1チーム1チャンネルを作成。(講師を含めて下さい。)
* **運用ルール:** チーム内の議論も極力Slack上で行って下さい。

* **顧客対応:**
* 顧客とのやり取りは個別に講師と行って下さい。
* その時、**『顧客として対応をお願いします。』**を文頭に入れて下さい。

### 3. 顧客への質問ルール

1. **「何に困っているか」を明確にする:**
* ×「どうすればいいですか？」
* ○「○○機能を実装するにあたり、A案とB案で迷っています。顧客としてはどちらが使いやすいですか？」


2. **自分の考え（仮説）を添える:**
* 「私たちは、使い勝手を優先して○○という仕様にしようと考えていますが、よろしいでしょうか？」という提案型にさせる。

3. **「専門用語」を使わない:**
* 顧客はITに詳しくない設定なので、「DBの正規化が〜」ではなく「データの整理の仕方が〜」といった言葉選びを意識させる。

#### QA運用の徹底

口頭やSlackでのやり取りだけで済ませず、必ず**「QA表」というエビデンス**を残すように指導します。

* **「仕様」の正解はQA表にある:** 顧客と合意した内容は、プログラムの正解となります。
* **即座に転記:** Slackで回答をもらったら、5分以内にQA表を更新すること。
* **一意のID管理:** 各質問に `Q001` などの番号を振り、コードのコメントやコミットメッセージに関連番号を添えられるようにします。
* **ルール:** QA表に載っていない仕様は、これを納品時にバグとして扱います。

### 4. テスト

* **テスト項目の作成:**
* 正常系（正しく動くケース）だけでなく、異常系（エラーが出るべきケース）の項目も作成して下さい。

* **テストの実施:**
* 単体テスト、結合テストを行い、期待通りに動くことを確認して下さい。
* 発見されたバグは「課題(T)」として表に追加し、修正・再テストを行って下さい。

## プログラミング総合演習：キックオフ指示書

> ### 演習の進め方とルール
> 
> 
> 1. **チーム結成・環境構築**
> 
> 
>       * GitHubリポジトリを作成し、講師を招待してください。
>       * Slackでチームチャンネルを作成し、講師を追加してください。(チーム内部用)
> 
> 
> 2. **要件のヒアリング（質問）**
> 
> 
>       * 要望書を読み込み、不明点はSlackで講師（顧客）に質問してください。
>       * ただし、質問は「提案型（〜にしようと思いますが良いですか？）」で行うこと。
> 
> 
> 3. **開発プロセス（Git運用）**
> 
> 
>       * `feature/`ブランチで作業し、完了したら`develop`へプルリクエストを送ること。
>       * 講師によるレビュー（またはリーダーの承認）を経てマージしてください。
> 
> 
> 4. **QA/課題表作成**
> 
> 
>       * QA表をテンプレートをもとに作成し、講師を招待してください。
>       * 進捗が確認できるように更新を徹底して下さい。
> 
> 
> 5. **製造**
> 
> 
>       * 設計に基づきコーディングを行います。
>       * チーム内で役割分担をし、随時Gitへコミット（履歴保存）してください。
> 
> 
> 6. **テスト**
> 
> 
>       * 作成したテスト項目に基づき動作確認を行ってください。
>       * すべてのテストが「OK」になるまで、修正と確認を繰り返します。
> 
> 
> 7. **納品**
> 
> 
>       * 最終的に`main`ブランチに統合し、タグ `v1.0.0` を付けて顧客に対し「納品完了」をSlackで報告してください。
>       * 報告時には、リポジトリのURLと、動作確認済みのQA/課題表も合わせて提出してください。
>       * リポジトリの doc/ ディレクトリに必要なドキュメントがすべて揃っていることを、報告前の最終チェック項目とします。
