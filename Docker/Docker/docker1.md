# **Dockerとファイアウォールの関係**

Docker環境におけるファイアウォールの管理は、通常のLinuxサーバーの設定とは異なる「罠」が存在します。実務で混乱を招きやすいポイントを整理しました。

## **1\. 基本原則：Dockerは「iptables」を直接操作する**

Linuxのファイアウォールの実体はカーネル内の **iptables（または nftables）** です。

Dockerをインストールして起動すると、Dockerは自前で **「DOCKER」という専用のチェーン（ルール群）** を iptables 内に勝手に作成します。

* **認識の確認**: 「Linuxの設定で行う」というのは正しいですが、人間が手動で設定する前に、**Dockerが自動で設定を書き換えてしまう** というのが実態です。

## **2\. 注意点：OS側のファイアウォール設定を「無視」することがある**

ここが最大の注意点です。Ubuntuなどで ufw（Uncomplicated Firewall）を使っている場合、以下のような現象が起こります。

* **現象**: ufw で「80番ポートを閉じている」つもりでも、Dockerで \-p 80:80 としてコンテナを起動すると、**外部からアクセスできてしまいます。**  
* **理由**: Dockerが作成する iptables ルールは、ufw などの管理ツールが作るルールよりも優先順位が高い場所に書き込まれるためです。

## **3\. 実務でのファイアウォール管理はどうすべきか？**

マルチコンテナ環境を構築・運用する際は、以下の3つのレイヤーで考えます。

| レイヤー | 管理方法 | 役割 |
| :---- | :---- | :---- |
| **クラウド/物理 FW** | AWS Security Group / Azure NSG 等 | **【最優先】** サーバーそのものにパケットが届く前に遮断する。最も安全。 |
| **Linux OS (Host)** | iptables / firewalld / ufw | Dockerとの競合に注意が必要。基本的にはDockerにお任せにする部分が多い。 |
| **Docker Network** | docker-compose.yml 内の networks | **【推奨】** コンテナ間の「内側」の通信を制限する。外部に公開しないコンテナはネットワークを分離する。 |

## **4\. セキュリティを高めるためのベストプラクティス**

今回の演習のような構成でセキュリティを高めるには、以下の設定が推奨されます。

1. **ホストのバインドを制限する**:  
   * ports: \["80:80"\] ではなく ports: \["127.0.0.1:80:80"\] と書くと、外部からの直接アクセスを禁止し、同じホスト内（あるいはリバースプロキシ経由）からのみアクセスを許可できます。  
2. **内部ネットワークの分離**:  
   * DBコンテナなどは、外部公開用のネットワークには参加させず、アプリコンテナだけが所属する「内部専用ネットワーク」に配置します。  
3. **Dockerにiptablesを触らせない設定（上級者向け）**:  
   * 設定でDockerによる自動操作をオフにできますが、ネットワーク設定をすべて手動で書く必要があり、非常に難易度が高いです。

## **まとめ**

* **Linuxの仕組みを使っている**: はい、その通りです。  
* **設定の主体**: 人間が書く設定よりも **Docker Engineが自動で書く設定が優先される** という点に注意が必要です。

そのため、実務では「Linuxのファイアウォールソフト」をいじるよりも、\*\*「クラウド側のファイアウォール（AWS等）」か「Docker Composeのネットワーク定義」\*\*で制御するのが一般的です。
