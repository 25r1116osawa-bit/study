# **Docker基本演習：仮想化技術 詳細解説資料**

## **1\. 仮想化とは何か？**

仮想化（Virtualization）とは、物理的なハードウェアリソース（CPU、メモリ、ストレージ、ネットワーク）を論理的に抽象化し、実際の構成にとらわれずに柔軟に分割・統合して利用する技術です。

かつては「1つの物理サーバに1つのOS」をインストールするのが一般的でしたが、この方法ではサーバの計算能力が余って無駄になったり、特定のソフトを入れるために別のサーバを買う必要があったりと、コストと柔軟性に課題がありました。仮想化はこれらの物理的な制約を取り払い、1台の強力なマシンの中で「あたかも複数台のマシンが動いているかのような」環境を実現します。

### **【深掘り】シミュレーション、エミュレーション、仮想化の決定的な違い**

これらは「何かを真似る」という点では共通していますが、そのアプローチと目的が異なります。

| 用語 | 動作の本質 | 主な目的と代償 | 具体的な活用シーン |
| :---- | :---- | :---- | :---- |
| **仮想化** | **物理資源の切り出し**。本物のハードウェア能力を、OSレベルで直接利用できるように分配する。 | **高効率・低負荷**。物理マシンに近い速度が出るが、ホストと異なるCPU命令は動かせない。 | Docker, クラウドサーバ (AWS/Azure), VMWare |
| **シミュレーション** | **「見た目」と「論理」の模倣**。ソフトウェア的に「こう動くはず」という計算を再現する。 | **分析と検証**。内部構造は本物と違っても、結果が合えば良い。実機がなくてもテストできる。 | 飛行機のフライトシミュレーター, 気象予測, 物理演算 |
| **エミュレーション** | **「構造」の完全再現**。別のハードウェア（CPUなど）をソフトウェアで1から作り直す。 | **完全な互換性**。本来動かないソフトが動くが、変換処理が重く、速度は大幅に低下する。 | ゲーム機エミュレーター, ARM搭載MacでのIntel用アプリ実行 |

#### **身近な例：スマートフォンの開発環境**

* **iOS Simulator（シミュレータ）**: MacのOS上で動作する「iOS風のアプリ」です。MacのCPUをそのまま使って高速に動きますが、実際のiPhoneのハードウェア（カメラや通信チップの細かい挙動）までは再現しません。  
* **Android Emulator（エミュレータ）**: PC上に「仮想的なARM CPU」を構築し、Androidスマホを丸ごと1台再現しようとします。そのため重いですが、実機に近い挙動が期待できます。

## **2\. 仮想化技術の3大分類**

**【参照：図1. 仮想化の3大分類 / 図2. 各方式の比較詳細】**

仮想化はその仕組み（アーキテクチャ）によって、パフォーマンスや導入のしやすさが大きく変わります。

### **① ホストOS型（手軽な実験室）**

WindowsやMacの上に、VirtualBoxなどの専用ソフトをインストールして仮想マシンを動かします。

* **メリット**: OSを消すことなく別のOSを試せるため、初心者の学習に最適です。  
* **デメリット**: 土台となる「ホストOS」と「仮想化ソフト」の両方を経由するため、動作は重くなります。

### **② ハイパーバイザー型（企業のデータセンター）**

WindowsなどのOSすら入れず、ハードウェアに直接「ハイパーバイザー」と呼ばれる管理ソフトを入れます。

* **メリット**: 余計なOSが介在しないため、非常に高速で安定しています。クラウドサービスの裏側はほぼこれです。  
* **デメリット**: 専用の管理知識が必要で、個人のPCでメイン環境として使うにはハードルが高いです。

### **③ コンテナ型（革新的な軽量化：Docker）**

これまでの仮想化が「OSごと入れ直す」のに対し、コンテナは「ホストOSの核（カーネル）を共有」します。

* **メリット**: 重いOSを起動させる必要がないため、**起動は数秒、メモリ消費もわずか**です。  
* **ポイント**: コンテナはOSではなく「隔離されたプロセス」として動きます。

## **3\. Dockerイメージとコンテナ：静と動の理解**

**【参照：図3. Dockerイメージとコンテナの関係】**

Dockerをマスターする鍵は、この2つの状態を明確に区別することにあります。

### **Dockerイメージ（不変の設計図）**

イメージは、アプリ実行に必要なすべてのファイル（ライブラリ、設定ファイル、コードなど）を詰め込んだ「スナップショット」です。

* **不変性 (Immutable)**: 一度作成したイメージは中身を書き換えることができません。これにより「昨日動いたのに今日動かない」という問題を根絶します。  
* **レイヤー構造**: イメージは「OS層」「Python層」「アプリ層」のように積み重なってできており、変更があった層だけを効率よく転送できます。

### **Dockerコンテナ（動作する実体）**

イメージという設計図を元に、メモリ上に展開された実行環境です。

* **独立性**: 同じイメージから100個のコンテナを立ち上げても、それぞれは独立しています。  
* **使い捨て**: コンテナは「使い捨て」が基本です。不調なら消して、新しいコンテナを数秒で作り直すのがDocker流の運用です。

## **4\. 「Python venv」と「Docker」：環境隔離の深度**

**【参照：図4. Python venv vs Docker】**

どちらも「環境を分ける」と言いますが、守備範囲が全く異なります。

| 比較項目 | Python venv | Docker (コンテナ) |
| :---- | :---- | :---- |
| **隔離の深さ** | 表面のみ（Pythonライブラリ） | 根底から（OS設定、ライブラリ、ミドルウェア） |
| **依存関係の解決** | Python以外のソフト（DB等）は別途インストールが必要 | DB、Webサーバ、設定すべてを1つに含める |
| **再現性の保証** | 「自分のPCでは動くが他人のPCでは壊れる」可能性がある | 「誰の、どのPCでも、全く同じバイナリ」が動く |
| **イメージ** | **部屋の模様替え**。床や壁（OS）は変えられない。 | **コンテナハウスごと搬入**。中身が完成した状態で届く。 |

## **5\. 学習環境（WSL2 × Docker）の舞台裏**

**【参照：図5. 学習環境（WSL2 × Docker）の舞台裏】**

現在、WindowsでDockerを使う際、内部では「Docker Desktop」と「WSL2」という2つの巨人が協力しています。

1. **司令塔としての Windows (Docker Desktop)**:  
   私たちが普段操作するGUI画面や、ターミナルで入力する docker コマンドを受け取ります。これは「管理ツール」としての役割です。  
2. **現場としての Linux (WSL2)**:  
   Dockerは本質的にLinuxの技術です。Windowsの中で「本物のLinux（WSL2）」を動かし、その中でコンテナを高速に動作させています。  
3. **魔法の架け橋**:  
   Docker Desktopの「WSL2 Backend」機能により、Windows側のファイルとWSL2側のシステムがシームレスに繋がり、私たちはWindowsを使いながらにしてLinuxの恩恵を100%享受できるのです。

## **6\. 疎結合な構成：コンテナ同士の連携**

Dockerの真骨頂は、1つのコンテナにすべてを詰め込むのではなく、\*\*「役割ごとにコンテナを分ける（疎結合）」\*\*ことにあります。

### **「全部入り」から「役割分担」へ**

既存のLAMP環境（Apache, PHP, MariaDB, Python）を移行する場合、以下のようにコンテナを分離して連携させます。

| コンテナ名 | 役割 | 連携方法 |
| :---- | :---- | :---- |
| **Webコンテナ** | Apache / PHP (WordPress) | フロントエンドとして80番ポートで待機 |
| **Appコンテナ** | Python (Flask) | Webコンテナからのリクエストを受け取り処理 |
| **DBコンテナ** | MariaDB | 各アプリからのデータ保存・抽出要求に応える |

### **コンテナを分ける3つのメリット**

1. **環境のクリーン化**: PythonのアップデートがApacheの設定に影響を与えるといった「依存関係の衝突」を防ぎます。  
2. **スケーラビリティ**: 「アクセスが増えたのでFlaskのコンテナだけ3つに増やす」といった柔軟な運用が可能になります。  
3. **保守性**: 各Dockerfileが短くシンプルになり、エラー箇所の特定が容易になります。

## **7\. なぜDockerが世界標準になったのか？**

1. **環境のポータビリティ**: 開発者のPCで作った「イメージ」をそのまま本番サーバへ持っていけば、必ず同じ挙動をします。  
2. **開発スピードの向上**: コマンド1つで、チーム全員が全く同じ開発環境を数分で手に入れられます。  
3. **リソースの有効活用**: 1台のサーバで多くのコンテナを動かせるため、コストを劇的に抑えられます。

## **8\. 究極の自動化へ：Kubernetes (k8s)**

Dockerで個々のコンテナの作り方を学んだ先には、大量のコンテナを自動管理する技術が待っています。

* **オートスケーリング**: 負荷に応じてコンテナを自動増減。  
* **セルフヒーリング**: 故障したコンテナを自動で検知して再起動。  
* **ローリングアップデート**: サービスを止めずに新バージョンへ切り替え。

まずはDockerという「個の技術」を完璧にマスターし、環境をコード化する便利さを体感しましょう。
