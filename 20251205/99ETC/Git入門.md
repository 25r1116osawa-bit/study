# Git入門講義ノート
Gitは、**ソフトウェア開発においてチームで協力し、効率的に作業を進めるための必須ツール**です。これから、Gitの基本から、チームで使うための実践的な流れまでをしっかり学んでいきましょう！🚀

## 💡 バージョン管理の必要性

まず、「**バージョン管理**」とは何か、そしてなぜそれが必要なのかを説明します。

皆さんは、レポートや資料を作成している時、うっかり重要な変更を消してしまったり、「前の状態に戻したい」と思ったりしたことはありませんか？

バージョン管理とは、**ファイルやソースコードの変更履歴を記録し、管理すること**です。

- **「バージョン」**：ファイルのある時点の状態のこと。
- **「管理」**：いつ、誰が、どんな変更を加えたかを記録し、必要な時にいつでもその時点の状態に戻せるようにすること。

### 🚨 Gitがないとどうなる？

バージョン管理がないと、例えば以下のような問題が起こります。

1. **ファイルの混乱**: 「`document_final_20251118.doc`」とか「`document_final_re.doc`」のように、似たようなファイル名が増えて、どれが最新で正しいファイルなのかわからなくなります。
2. **変更の取り消しが困難**: 間違った変更を元に戻すのが大変、または不可能になります。
3. **共同作業の衝突**: チームで同じファイルを編集すると、誰かの変更が別の人の変更で上書きされてしまう（**コンフリクト**）可能性が高まります。

### 🌟 Gitの役割

Gitは、これらの問題を解決するための**分散型バージョン管理システム**です。

- **分散型（Distributed）**：各開発者（皆さん）のパソコン（ローカル環境）にも、変更履歴の完全なコピーが保存されます。これにより、インターネット接続がなくても作業ができ、万が一サーバーに問題があっても安心です。

#### subversionやCVSとの違い
- Gitは**高速**で、**ブランチ操作が軽量**です。これにより、新しい機能の開発やバグ修正を独立して行い、後で簡単に統合（マージ）できます。
- Gitは**変更内容の差分管理**に優れており、どの部分がどのように変わったかを詳細に追跡できます。
- Gitは**オープンソース**であり、世界中の開発者が参加しており、豊富なツールやサービス（例：GitHub, GitLab）が利用可能です。
- subversionは中央集権型であり、サーバーがダウンすると作業ができなくなるリスクがあります。
- subversionはブランチ操作が重く、頻繁にブランチを切る開発スタイルには向いていません。
- subversionは変更履歴の管理がGitほど詳細ではなく、差分の追跡が難しい場合があります。
- subversionはオープンソースではありますが、Gitほどの活発なコミュニティやエコシステムはありません。

## 💻 Gitのインストールと初期設定

Gitを使うには、まず皆さんのパソコンにGitをインストールし、\*\*「私が誰であるか」\*\*をGitに教えてあげる必要があります。

### 1\. インストール

皆さんが使っているOS（Windows, macOS, Linux）に応じて、公式サイトからGitをダウンロードしてインストールしてください。

### 2\. 初期設定（ユーザー登録）

インストール後、\*\*コマンドライン（ターミナル）\*\*という文字だけで操作する画面で、以下のコマンドを実行します。

```bash
# 自分の名前を設定
$ git config --global user.name "Your Name"

# 自分のメールアドレスを設定
$ git config --global user.email "your.email@example.com"
```

- **`git config`**: Gitの設定を変更するためのコマンドです。
- **`--global`**: このパソコンの**全て**のGitプロジェクト（後述の**リポジトリ**）にこの設定を適用するという意味です。
- この情報が、皆さんが行った\*\*変更履歴に「記録」\*\*されます。

## 📂 ローカルリポジトリの作成

いよいよ、Gitで管理する「箱」を作ります。この箱のことを**リポジトリ**（Repository）と呼びます。

- **リポジトリ**: ファイルの変更履歴（バージョン）を格納する**データベース**のようなものです。

### 1\. プロジェクトフォルダを作成

まず、作業したいフォルダ（例：`my-project`）をパソコン上に作ります。

### 2\. リポジトリの初期化

そのフォルダ内で、以下のコマンドを実行します。

```bash
cd my-project # フォルダに移動
git init      # Gitの管理を開始
```

- **`git init`**: 「このフォルダをGitで管理するぞ！」と宣言し、Gitが履歴を保存するための隠しフォルダ（`.git`フォルダ）を作成します。これで**ローカルリポジトリ**が作成されました。

## 🌐 GitHubの基本操作

Gitは皆さんのパソコン（ローカル）で動くツールですが、**チームで共有したり、バックアップを取ったりする**ためには、インターネット上にあるサービスを使います。その代表が\*\*GitHub（ギットハブ）\*\*です。

GitHubは、**クラウド上**にあるGitリポジトリのホスティングサービスで、世界中の開発者が使っているプラットフォームです。

### 1\. アカウント作成

GitHubのウェブサイトにアクセスし、メールアドレスなどを登録してアカウントを作成します。（これは通常のWebサービスと同じ手順です。）

### 2\. リモートリポジトリの作成

GitHubのサイト上で「New repository」などのボタンから**リモートリポジトリ**（遠隔地のリポジトリ）を作成します。

- ここでは、リポジトリ名を設定するだけでOKです。

## 🤝 チームでのリポジトリ共有と作業の流れ

ここがGitの最も重要なポイントです。皆さんがこれから行う、チームでの一般的な開発の流れを見てみましょう。

### 1\. リポジトリのクローン（取得）

チームの誰かがGitHubにリポジトリを作成したら、他のメンバーはそのリポジトリを自分のパソコンに**コピー**（取得）します。

```bash
git clone [GitHubリポジトリのURL]
```

- **`git clone`**: リモートリポジトリの全てのファイルと履歴を、ご自身のローカルリポジトリとして丸ごとコピーするコマンドです。

### 2\. 変更・コミット・プッシュの流れ

皆さんがコードの変更を加えた後、Gitにその履歴を記録し、GitHubに共有するまでの流れは以下の3ステップです。

| ステップ | コマンド | 役割 | 専門用語の説明 |
| :--- | :--- | :--- | :--- |
| **Stage** (ステージング) | `$ git add [ファイル名]` | どのファイルを今回の変更として記録するかを**選択**する作業です。 | **インデックス/ステージングエリア**: 作業ディレクトリとリポジトリの間に存在する「一時保管場所」です。ここにファイルを追加することで、次のコミットの対象としてマークされます。 |
| **Commit** (コミット) | `$ git commit -m "変更内容の要約"` | ステージングされた変更を**ローカルリポジトリに正式に履歴として記録**します。 | **コミット**: 変更履歴の**最小単位**。誰が、いつ、どんな目的で変更したかを記録します。`"-m"` の後に付けるメッセージ（コミットメッセージ）は**変更内容を簡潔に**書きます。 |
| **Push** (プッシュ) | `$ git push origin main` | ローカルリポジトリに記録されたコミット履歴を、**リモートリポジトリ（GitHub）に送信して共有**します。 | **Push**: ローカルの変更をリモートに反映させる操作です。`origin`はリモートリポジトリの別名、`main`は通常、メインのブランチ名です。 |

### 3\. 他の人の変更を取得（プル）

チームメンバーが変更をGitHubにプッシュしたら、自分の作業を始める前に、最新の状態を取り込む必要があります。

```bash
git pull origin main
```

- **`git pull`**: リモートリポジトリの最新の変更を**取得**し、自分のローカルのファイルに**結合**（マージ）するコマンドです。これは「`git fetch`」と「`git merge`」の二つの操作を一度に行います。

## 🔄 マージ操作（変更の統合）

チーム開発では、通常、**ブランチ**（Branch）という機能を使って作業します。

- **ブランチ**: メインの作業の流れ（**main/masterブランチ**）から枝分かれした、**独立した作業空間**です。これにより、メインのコードを壊すことなく、新しい機能の開発やバグ修正が行えます。

自分のブランチで作業が完了したら、その変更をメインの流れに戻す必要があります。これが\*\*マージ（Merge）\*\*です。

### マージとは？

**マージ**：異なるブランチで行われた変更を**一つのブランチに統合**することです。

もし、同じファイルの**同じ行**を、自分と他の人が同時に変更していて、どちらの変更を残すべきかGitが自動で判断できない場合、Gitは\*\*コンフリクト（Conflict / 衝突）\*\*を報告します。

### 衝突の解決

コンフリクトが発生したら、開発者である皆さんが手動でファイルを開き、「どちらの変更を採用するか」「両方の変更をどう組み合わせるか」を判断し、ファイルを修正します。

コンフリクトを解決してファイルを`git add`、`git commit`すれば、マージが完了します。

## 📝 本日のまとめ

| 用語 | 意味 |
| :--- | :--- |
| **Git** | 分散型バージョン管理システム。変更履歴を管理するツール。 |
| **リポジトリ** | 変更履歴が格納される「箱」（データベース）。ローカルとリモートがある。 |
| **GitHub** | リモートリポジトリをホスティングするウェブサービス。 |
| **コミット** | 変更をローカルリポジトリに正式に記録する最小単位の操作。 |
| **プッシュ** | ローカルのコミットをGitHubに送信し、チームと共有する操作。 |
| **プル** | GitHubの最新の変更を取得し、ローカルに統合する操作。 |
| **ブランチ** | 独立した作業空間。 |
| **マージ** | 異なるブランチの変更を統合する操作。 |
| **コンフリクト** | マージ時にGitが自動で統合できず、開発者の判断が必要になる「衝突」。 |

これがGitとGitHubを使ったチーム開発の基本的な流れです。
